# 手動順番ワークフロー・全体ランキング・所要データに関する議論まとめ（2026-03-24）

## 目的

このドキュメントは、現場リーダー向けの優先順位付けワークフローについて、今回の議論で出た論点をまとめたものです。

対象は次の3領域です。

- 手動順番画面のUI/運用改善
- 全体ランキングの現状評価
- `FSIGENSHOYORYO`（所要・総分）と、今後の個数/実績工数突合の位置づけ

## 1. 現在の業務ワークフロー認識

現時点の想定ワークフローは次のとおり。

1. 生産管理部署発信の CSV で、製番単位の納期日を把握する。
2. 納期管理画面で、各製番に対して表面処理ごとの納期を一括設定する。
3. 本来はここで全体ランキングを生成して全工程へ配りたいが、現時点では有意な結果が出にくい。
4. そのため手動順番画面へ戻り、各資源CDごとに対象製番を検索条件へ入れ、item を全表示する。
5. 各資源CDで順番を手動で付ける。
6. ただし、資源CDごとの局所最適を積み上げても、全工程ではコンフリクトが起きる。

重要な認識:

- 今の `手動順番` は「自動順位の微修正UI」ではない。
- 実態としては、**全体ランキングが未成熟な間、人間が全工程を見ながら順番を組み立てる主作業面**である。

## 2. 手動順番画面に対する率直な所見

### 現状の強み

- 上ペインで工場/端末/資源CDの俯瞰ができる。
- 下ペインで既存の生産スケジュール一覧UIを再利用できる。
- 単一資源CDに絞る制約により、誤編集をある程度防げている。

### 現状の根本課題

- 各資源CD内の順番は決められるが、**工程間の整合をその場で見ながら決める支援が弱い**。
- 編集操作が「絶対順位を全部決める」前提に近く、認知負荷が高い。
- 上ペインの俯瞰と下ペインの編集がつながっておらず、「全体を見て、どこをどう直すか」に飛びにくい。
- 全体ランキングが未成熟なため、手動順番に頼るしかないが、手動順番のUIがまだ「作業台」として十分ではない。

## 3. UI改善案として出たもの

### 3-1. 基本方針

手動順番画面は「順位入力画面」ではなく、**全工程の順序を段階的に組み立てる作業台**として扱うべき。

### 3-2. 絶対順位入力を主役にしない

今のように `processingOrder` を直接全部埋めるより、主操作は相対操作へ寄せる。

例:

- `先頭へ入れる`
- `この行の後ろへ入れる`
- `このブロックをまとめて前へ`
- `未確定へ戻す`

考え方:

- 人間は「何番か」を全部考えるより、「これは先にやる/後ろでよい」を決める方が得意。
- 番号は最後にシステムが連番化すればよい。

### 3-3. 帯分けを下書きとして使う

帯分けは最終成果物ではない。  
**確定順位を作る前段の下書き**として使う。

想定する流れ:

1. まず item を帯ごとに並べる
2. 帯の中だけを手動で整える
3. 最後に資源CDごとに `1..N` を確定する

帯の例:

- 帯A: かなり優先して見たい
- 帯B: 早めに処理したい
- 帯C: 余裕がある
- 保留: 情報不足や判断待ち

### 3-4. 帯分けからオペレーター向け確定順位へ落とす

帯分けで止めず、最後は各資源CDごとに線形順序へ落とす。

つまり:

- 帯Aの先頭 = 1番
- 帯Aの次 = 2番
- ...
- 帯Bの先頭 = A帯の末尾の次

これで、最終的には各オペレーターに今まで通り `processingOrder` 相当の順番を提示できる。

### 3-5. 工程間影響の可視化

理想的には、ある資源CDで item を動かしたときに、その製番の他工程の状況も見えるべき。

見せたいもの:

- 前工程/後工程での位置
- 同じ製番の他工程との整合崩れ
- 下流待ちの発生可能性
- 全体の中での大まかな位置

ただし現時点では、コンフリクトの赤判定基準が弱い。  
よって、すぐに「赤バッジで強判定」へ行くより、まずは **共通の粗い優先軸を作る** 方が先。

### 3-6. UIデザインプレビュー

この議論を視覚化した静的プレビューを追加済み。

- `docs/design-previews/manual-order-band-to-operator-rank-preview.html`

このプレビューは、

- 上ペイン: 編集対象の選択
- 中央: 帯分け → 帯内調整
- 右: オペレーター向け最終順位

という1画面の流れを表す。

## 4. 帯分けアルゴリズムについての結論

### 4-1. 納期 + 残工程数だけでは弱い

議論の中で一度は

- 納期
- 残工程数

の2軸で帯を作る案を検討したが、これは弱いという結論になった。

理由:

- 同納期・同残工程でも、案件の重さが違う
- 数量差や加工負荷の差が見えない

### 4-2. `FSIGENSHOYORYO` を加えた3軸は有望

現時点では、帯算出の初版は

- 納期
- 残工程数
- `FSIGENSHOYORYO`（所要・総分）

の3軸が現実的。

ただし `FSIGENSHOYORYO` は生値でそのまま使わない。

## 5. 実DBの `FSIGENSHOYORYO` 分析結果

分析対象:

- dashboard: `3f2f6b0e-6a1e-4c0b-9d0b-1a4f3f0d2a01`
- winner 行のみ

主な結果:

- 行数: `16,767`
- 数値化率: `100%`
- 行単位分布:
  - `p50=75`
  - `p75=180`
  - `p90=442`
  - `p95=630`
  - `max=28,800`
- 製番単位の未完総所要:
  - `842` 製番
  - `p50=1,920`
  - `p75=5,137.75`
  - `p95=11,377.95`
  - `max=33,900`
- 未完工程数との相関: `0.6757`
- 資源CDごとの差:
  - `p50` は `25〜360`
  - `p90` は `84〜1,200`

結論:

- `FSIGENSHOYORYO` は帯算出に使える
- ただし **ロングテールが強い**
- さらに **資源CDごとの差が大きい**
- よって、全資源共通の固定閾値は危険

詳細:

- `docs/analysis/production-schedule-fsigenshoyoryo-analysis-20260324.md`
- `docs/analysis/production-schedule-fsigenshoyoryo-profile.sql`

## 6. `FSIGENSHOYORYO` の使い方の判断

### 6-1. 現時点で言えること

- `FSIGENSHOYORYO` は「参考程度」で終わるほど弱くはない
- ただし「精密な時間」ではなく、**粗い負荷信号**として使うべき

### 6-2. 初期の使い方

#### 行単位（資源CD内）

- 資源CDごとの分位点で帯化する
  - 例:
    - `<= p50`: 軽
    - `p50 - p75`: 中
    - `p75 - p90`: 重
    - `> p90`: 特重

#### 製番単位

- `unfinished_total_required` を製番単位で集計し、分位点で帯化する

### 6-3. 避けるべき使い方

- 生値をそのまま加重平均へ突っ込む
- 全資源共通で「300分以上は重い」のように切る
- `残工程数` と同じ意味で二重加点する

## 7. 納期データ coverage の問題

実DB snapshot では、

- current 製番数: `842`
- `ProductionScheduleSeibanDueDate` と join できたのは `6`

だった。

所見:

- 「納期 + 所要 + 残工程数」で帯を切る方針は正しい
- ただし、**今の snapshot では納期軸の coverage が弱い**
- 納期入力 coverage を上げることが重要

## 8. 個数についての結論

今回の議論で最も重要な合意の1つ。

### 8-1. 個数は必須

理由:

- 今の `FSIGENSHOYORYO` は「総所要」であり、**負荷量**は見えるが **単位あたりの重さ**は見えない
- 同じ `資源CD × FHINCD` でも、`FSEIBAN` ごとに差が出る理由が
  - 個数差
  - 段取り差
  - 難易度差
  - 入力誤差
  のどれなのか切り分けられない

### 8-2. 個数が入るとできること

- `1個あたり所要 = FSIGENSHOYORYO / 個数`
- 同じ `資源CD × FHINCD` での `FSEIBAN` ごとのばらつき確認
- 品番/部品カテゴリごとのモデル工数候補
- `FSIGENSHOYORYO` の信頼性評価

率直な結論:

- 個数なしの `FSIGENSHOYORYO`: **帯補助には使える**
- 個数ありの `FSIGENSHOYORYO`: **モデル工数・ランキング根拠候補に昇格する**

## 9. 実績工数との突合で得られる知見

数年分の実績工数には

- 個数
- 全個数完了に費やした実績工数

があるため、将来的にかなり強い突合が可能。

突合で得られる知見:

- `FSIGENSHOYORYO` と実績総工数の相関
- `FSIGENSHOYORYO / 個数` と `perPieceMinutes` の相関
- 同じ `資源CD × FHINCD` で 1個あたり時間が安定しているか
- 数量依存なのか、難易度依存なのか
- 品番ベースのモデル工数が作れるか
- `所要` の信頼度スコアが作れるか

率直な判断:

- 個数追加後の突合分析は、**最優先でやる価値がある**
- これで初めて、`所要` をどこまで信じてよいかを根拠付きで判断できる

## 10. 現時点の総合結論

### 10-1. いまやるべきこと

1. `FSIGENSHOYORYO` を粗い帯補助として使う
2. ただし資源CD内分位点で扱う
3. UI上は「帯分け → 帯内調整 → 連番確定」の流れを目指す

### 10-2. 次にやるべきこと

1. 生産日程へ個数を追加する
2. `1個あたり所要` を計算可能にする
3. 実績工数と突合し、`資源CD × FHINCD` ごとの安定性を見る

### 10-3. その先

1. モデル工数の候補を作る
2. 帯算出を高度化する
3. 全体ランキングへ段階的に反映する
