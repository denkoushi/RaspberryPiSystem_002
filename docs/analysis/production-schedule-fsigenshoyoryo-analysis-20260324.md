# 生産日程 `FSIGENSHOYORYO` 分析（2026-03-24）

## 目的

- 生産日程 CSV の `FSIGENSHOYORYO`（所要・総分）が、帯算出に使えるかを評価する。
- 使える場合、どういう切り方・正規化なら活きるかを見極める。
- 既存の「参考程度」扱いから一歩進め、**どこまで信じてよい信号か**を明確にする。

## 実施方法

- 対象 DB: Pi5 本番 DB `borrow_return`
- 対象 dashboard: `3f2f6b0e-6a1e-4c0b-9d0b-1a4f3f0d2a01`
- 対象行: `buildMaxProductNoWinnerCondition()` と同等の winner 行のみ
  - キー: `FSEIBAN + FHINCD + FSIGENCD + FKOJUN`
  - tie-break: `ProductNo desc -> createdAt desc -> id desc`

再実行用 SQL は [`production-schedule-fsigenshoyoryo-profile.sql`](./production-schedule-fsigenshoyoryo-profile.sql) を参照。

## 結論

### `FSIGENSHOYORYO` は使えるか

**使える。しかも帯算出にはかなり有望。**

ただし条件付き:

- 生値をそのまま強く使わない
- 全資源共通の閾値で切らない
- 外れ値をクリップする
- `残工程数` の補助以上・絶対順位の決定未満として使う

### どう使うと活きるか

#### 1. 行単位（手動順番・資源CD内）

使い方:

- `FSIGENSHOYORYO` を **資源CDごとの分位点**で帯化する
  - 例:
    - `<= p50`: 軽
    - `p50 - p75`: 中
    - `p75 - p90`: 重
    - `> p90`: 特重

理由:

- 同じ `180分` でも、資源によっては普通、別資源では重い。
- 資源CD内の比較なら、`FSIGENSHOYORYO` はかなり素直に効く。

#### 2. 製番単位（全体の粗い帯）

使い方:

- `unfinished_total_required` を製番単位で集計し、全製番の分位点で帯化する
  - 例:
    - `<= p25`: 軽い製番
    - `p25 - p50`: 中
    - `p50 - p75`: 重
    - `> p75`: 特重

理由:

- `p50=1920`, `p75=5137.75` と製番単位の差が大きく、粗い帯で見る価値が高い。
- 全体ランキング未完成の段階でも「重い製番群」を前倒しで見る補助になる。

#### 3. 残工程数との併用

推奨:

- `残工程数` は独立信号として残す
- ただし `所要` と同時に強く加点しすぎない

初期方針:

- `納期`: 主軸
- `所要`: 第2軸
- `残工程数`: 補正軸

## 初期の帯算出方針（提案）

### 行単位の帯

1. 納期で大枠
   - overdue / 今日明日 / 3日以内 / それ以降 / 未設定
2. 同じ納期帯の中で、資源CD内 `FSIGENSHOYORYO` 分位点で重さを付ける
3. 残工程数が多いものを1段だけ前寄せ

### 製番単位の帯

1. 納期帯
2. `unfinished_total_required` の分位点
3. 残工程数（未完工程数）の分位点

## 非推奨

- 全資源共通の「300分以上は重い」のような固定閾値
- 生の `FSIGENSHOYORYO` をそのまま加重平均へ突っ込む
- `所要` と `残工程数` を同じ意味で二重加点する

## 次の一手

1. 納期入力の coverage を上げる
2. 行単位帯に `資源CD内分位点` を採用する
3. 製番単位帯に `unfinished_total_required` を採用する
4. 後段で、部品名・品番ごとのモデル工数と比較して教師信号として使えるか検証する
